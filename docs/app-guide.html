App Guide

For app developers adding PingIt to an Android app: how to install the SDK, configure it, and call it. To run the backend and dashboard, see the Dashboard Guide.

Get started

1. Get an appId

Register your app in the dashboard (or ask an operator to). You receive an appId and a one-time API key. The appId identifies your app on every request and scopes your data.

2. Add the SDK

The SDK lives under sdk-android/. Until it is published to a Maven repository, add it as a local module or composite build, or build the AAR and depend on it:

cd sdk-android
./gradlew :android:assembleRelease    # outputs android/build/outputs/aar/

Requirements: Android Studio, JDK 17, minSdk 24. The library declares the INTERNET and ACCESS_NETWORK_STATE permissions for you.

3. Configure auto-initialization (recommended)

PingIt initializes itself on app startup using AndroidX App Startup. You do not need to call any init code. Add your configuration as meta-data inside the <application> element of your AndroidManifest.xml:

<application ...>
    <meta-data android:name="com.pingit.APP_ID"   android:value="app_xxxxxxxx" />
    <meta-data android:name="com.pingit.ENDPOINT" android:value="https://api.example.com" />

    <!-- optional -->
    <meta-data android:name="com.pingit.HISTORY_MODE"      android:value="LAST_LOCAL" />
    <meta-data android:name="com.pingit.DATA_SAVER"        android:value="false" />
    <meta-data android:name="com.pingit.FALLBACK_ENDPOINT" android:value="https://backup.example.com" />
</application>

After startup, get the ready-to-use client anywhere in your app:

val pingit = PingIt.getInstance()

Initialize in code instead (optional)

To configure the SDK at runtime, disable the startup initializer in your manifest and call init yourself:

<provider
    android:name="androidx.startup.InitializationProvider"
    android:authorities="${applicationId}.androidx-startup"
    tools:node="merge">
    <meta-data android:name="com.pingit.android.startup.PingItInitializer"
               tools:node="remove" />
</provider>
val pingit = PingIt.init(
    appId = "app_xxxxxxxx",
    config = Config(
        endpoint = "https://api.example.com",
        historyMode = HistoryMode.LAST_LOCAL,   // NONE | LAST_LOCAL | SERVER_HISTORY
        dataSaver = false,
        fallbackEndpoint = null,
    ),
    context = applicationContext,
)

Core functions

FunctionDescription
PingIt.getInstance()The client created by auto-initialization.
PingIt.init(appId, config, context)Initialize in code; returns and registers the client.
isReadyFor(profile)Measure and return pass or fail with a reason.
runTest()Run a full measurement; returns metrics, score, and label.
ping()Round-trip latency in milliseconds.
getHistory(limit)Recent results (depends on the history mode).
refreshProfiles()Force a refresh of the readiness thresholds.
cancel()Cancel an in-flight measurement.

All measurement functions are suspend functions; call them from a coroutine.

Examples

Ask a readiness question (the common case)

val ready = PingIt.getInstance().isReadyFor(Profile.VIDEO_CALL)
if (!ready.passed) {
    // ready.reason -> "latency too high" / "upload too low" / "no connection"
    showWarning("Your connection may not handle a video call: ${ready.reason}")
}

isReadyFor returns ReadinessResult(profile, passed, reason, measured). Offline is handled honestly: with no connection you get passed = false, reason = "no connection", measured = null, never a fake pass.

Available profiles: MESSAGING, WEB_BROWSING, MUSIC_STREAMING, VOICE_CALL, VIDEO_CALL, HD_STREAMING, UHD_4K_STREAMING, CLOUD_GAMING, LIVE_BROADCAST, LARGE_UPLOAD.

Raw measurement and history

val client = PingIt.getInstance()
val result = client.runTest()
// result.downloadMbps, result.uploadMbps, result.latencyMs,
// result.jitterMs, result.score (0-100), result.label

val latency = client.ping()                  // latency only, in ms
val recent  = client.getHistory(limit = 20)  // depends on the history mode
client.refreshProfiles()                     // force a threshold refresh
client.cancel()                              // abort an in-flight test

History modes

ModeBehavior
NONEKeep nothing.
LAST_LOCAL (default)Keep only the most recent result on the device.
SERVER_HISTORYAlso upload each result to the server, buffered and retried.

Data saver

Set DATA_SAVER to true (or wire it to the operating system data-saver flag) to run smaller payloads and shorter tests for users on metered plans.

What the SDK does automatically

  • Profiles are fetched once and cached, then refreshed about once a day at a randomized time and whenever the version changes. Offline, it falls back to the cache and then to bundled defaults.
  • Uncaught exceptions are persisted immediately and uploaded on the next launch through WorkManager, so a crash before the app closes is not lost.
  • If the primary server is unreachable, measurement falls back to a public endpoint when one is configured. The readiness decision is always made on the device.

Try the demo

The demo/ folder is a small web app ("Hub") that shows the SDK behavior in everyday actions (message, video call, watch, cloud gaming, backup). It calls the server the same way the SDK does and uses the same scoring and evaluation logic, which makes it a quick way to see the value end to end without an emulator. See demo/README.md.

Build and test the SDK from source

The SDK logic lives in a pure-Kotlin core module that unit-tests on the JVM with no Android SDK required:

cd sdk-android/core
JAVA_HOME=/opt/homebrew/opt/openjdk@17 ./gradlew test

The Android library and sample app build in Android Studio (set sdk.dir in local.properties):

cd sdk-android
./gradlew :sample:assembleDebug     # the sample app
./gradlew :android:assembleRelease  # the SDK AAR
