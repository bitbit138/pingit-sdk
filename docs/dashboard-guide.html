Get started (run locally)

Prerequisites

ToolVersionNotes
Node22 or newerserver, dashboard, contracts
pnpm10 or newercorepack enable activates the pinned version
Docker and Composeany recentruns Postgres (no local Postgres needed)

Setup

corepack enable
pnpm install            # installs all workspace deps
pnpm build:contracts    # builds @pingit/contracts and emits profiles.json

@pingit/contracts is the single source of truth for shared types and the profile thresholds. Rebuild it after editing any shared type or threshold; pnpm gen:profiles also re-emits profiles.json into the Android core resources.

Database

pnpm db:up                                             # Postgres in Docker (:5432)
cp packages/server/.env.example packages/server/.env   # server config
pnpm db:migrate                                        # create tables and indexes
pnpm db:seed                                           # profiles, admin, and demo apps

db:seed prints the demo appId and creates the web demo's fixed app_demo. Default accounts:

  • Admin: admin@pingit.dev / ChangeMe123!
  • Developer: dev@pingit.dev / ChangeMe123!

The test suite uses a separate pingit_test database (auto-migrated). Create it once if needed: docker compose exec -T postgres createdb -U pingit pingit_test.

Run

pnpm dev            # builds contracts, then runs server (:8080) and dashboard (:5173)
pnpm dev:server     # server only
pnpm dev:portal     # dashboard only (proxies API calls to :8080)

Test

pnpm test                              # contracts, server, dashboard
pnpm --filter @pingit/server test      # API integration tests (Postgres up)
pnpm --filter @pingit/portal test      # dashboard unit tests
pnpm smoke                             # end-to-end HTTP flow against a running server

Deploy to production

Option 1: Docker Compose (single host)

The Compose file bundles Postgres and the server. The server image runs migrations and seed on boot through RUN_MIGRATIONS and RUN_SEED:

pnpm docker:up          # docker compose --profile full up --build -d
curl -s localhost:8080/health
pnpm docker:down

Option 2: Managed database and a container platform

  1. Provision Postgres and set DATABASE_URL.
  2. Build the image: docker build -f packages/server/Dockerfile -t pingit-server .
  3. Initialize once with RUN_MIGRATIONS=true RUN_SEED=true (or run pnpm db:migrate && pnpm db:seed as a one-off job), then run normally.
  4. Deploy on your platform (Fly, Render, ECS, a VM), exposing port 8080 behind HTTPS.

Serving the dashboard

  • Same origin (recommended): pnpm --filter @pingit/portal build and serve packages/portal/dist from the same host as the API, leaving VITE_API_BASE empty.
  • Separate static host: set VITE_API_BASE to the server URL at build time. The server allows cross-origin requests.

Operational notes

  • Health: GET /health is unauthenticated. Use it for load balancers and the SDK's fallback probe.
  • Scaling: measurement traffic is bandwidth-bound and stateless and can be hosted separately from the data API later. The default rate-limit store is in-memory; add a shared store such as Redis for multiple instances.
  • Backups: profiles, apps, and users are the critical configuration tables; results and crashes are append-only history.

Use the dashboard

Open the dashboard (default http://localhost:5173 in development) and sign in.

Roles

  • Admin: edit profile thresholds; see global analytics, health, and crash reports across all apps.
  • Developer: register apps to get an appId, and see analytics for their own app only. Access is scoped by role and by appId.

Common tasks

  • Register an app: in Apps and keys, register an app, then copy the appId and the one-time API key (not shown again). Give the appId to the app developer for PingIt.init or the manifest configuration.
  • Tune a profile: open Profiles, pick a profile, edit the metric minimum and maximum thresholds, and save. This bumps the global version, and SDKs pick up the new thresholds on their next refresh with no app release needed.
  • Read analytics: the Dashboard and Analytics pages show pass rate per profile, average latency over time, and test counts. KPI cards show tests today, pass rate, average latency, and fallback rate. The web demo reports under app_demo, so its runs appear here.
  • Investigate issues: the Crashes page lists uploaded crash and error reports per app.

How identity works

  • appId identifies the app. It is set once in the SDK configuration, much like a Stripe or Firebase key.
  • deviceId is an anonymous UUID the SDK generates on first run and stores on the device. It survives restarts and identifies one install over time. There are no hardware identifiers and no personal data. A reinstall starts a fresh deviceId.
